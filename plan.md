# 棋类游戏平台（C++）实施计划

## 1. 问题与目标
在现有 C++ 项目中实现一个可扩展的本地棋类游戏平台，首批支持：
- 五子棋
- 中国象棋

并具备：
- 内置 AI 对手（可选难度）
- 对局历史记录保存与回放
- 统一的游戏入口与规则引擎框架，便于后续扩展更多棋类

## 2. 总体方案
采用“通用框架 + 分游戏实现 + 分层 AI + 持久化模块”的架构：

1. **核心框架层**
   - 定义统一接口：`IGame`, `IGameState`, `IMoveGenerator`, `IPlayer`, `IAgent`
   - 统一回合控制、胜负判定入口、落子合法性校验入口
   - 统一序列化接口（用于存档/历史）

2. **游戏实现层**
   - `gomoku/`：15x15 棋盘、禁手规则（可配置开关）、连五判定
   - `xiangqi/`：完整走法规则、将军/应将判定、胜负和局判定

3. **AI 层**
   - 五子棋：启发式评估 + Minimax/Alpha-Beta（支持迭代加深）
   - 中国象棋：Alpha-Beta + 局面评估（棋子价值 + 位置表）
   - 难度通过搜索深度、候选步宽度、评估噪声控制

4. **数据与历史层**
   - 统一对局记录结构：元数据 + 步骤序列 + 结果
   - 本地存储（JSON/二进制二选一，默认 JSON，便于调试）
   - 支持查询历史列表、加载回放、继续残局

5. **交互层**
   - 首版提供 CLI 菜单与棋盘渲染（快速可用）
   - 后续可接图形前端（Qt/SDL）而不改核心规则与 AI

## 3. 执行清单（Todos）
1. **project-restructure**：重构目录与构建脚本，建立 core/ai/games/storage/ui 模块骨架。  
2. **core-engine**：完成通用回合引擎、状态机、统一接口与错误处理。  
3. **gomoku-module**：实现五子棋规则、棋盘、判定、CLI 对局流程。  
4. **xiangqi-module**：实现中国象棋规则、棋子走法、将军判定与对局流程。  
5. **ai-framework**：实现通用搜索框架与难度分级配置。  
6. **ai-gomoku-xiangqi**：接入五子棋/象棋 AI，并完成强度分档。  
7. **history-storage**：实现对局保存、读取、回放、继续对局。  
8. **unified-menu**：实现统一入口（选择游戏、难度、人机/双人、历史记录）。  
9. **tests-and-benchmark**：补充规则单元测试、对局回归测试、AI 性能基准。  

## 4. 关键设计约束
- **可扩展性优先**：新增棋类不应改动核心框架。
- **规则正确性优先于 AI 强度**：先保证规则完整与可测，再迭代 AI。
- **存档向前兼容**：记录格式加版本号，支持未来字段扩展。
- **性能目标**：AI 响应可配置；低难度即时响应，中高难度允许更长思考。

## 5. 待你确认后再实现的决策点
1. 首版交互是否先固定为 CLI（建议），图形界面后续迭代。
2. 历史记录格式是否接受 JSON（建议首版 JSON，后续可加二进制压缩）。
3. AI 难度档位数量（建议 3 档：简单/中等/困难）。

## 6. 里程碑交付顺序
1. 框架与五子棋先落地（最快拿到可玩版本）。  
2. 接入象棋与其 AI。  
3. 完成历史记录、回放与全量测试。
